## Service-Oriented Architecture (SOA) → 서비스로 이루어진 구조
- 서비스 지향 아키텍처(SOA)는 상호작용하는 소프트웨어 에이전트 간의 느슨한 결합(loose coupling)을 달성하는 것을 목표로 하는 **아키텍처 스타일**입니다.
- **모놀리식 디자인**(일체형 설계) 대신 개별 서비스(discrete services)에 초점을 맞춘 아키텍처 스타일입니다.

## 속성

- **반복 가능한 비즈니스 활동을 논리적으로 표현**하며, 특정한 결과를 제공하도록 설계됩니다.
- 스스로 온전하다.
- 소비자(consumer)에게는 **블랙 박스**와 같습니다. 소비자는 서비스의 내부 작동 방식에 대해 알 필요가 없습니다.
- 다른 서비스들로 구성될 수 있다.

- **Service Provider (서비스 제공자)**
    - 웹 서비스를 만들어내고, 서비스 레지스트리에 제공한다.
- **Service broker, service registry or service repository**
    - 주된 기능은 웹 서비스에 관한 정보를 **잠재적인 요청자**가 이용할 수 있도록 제공하는 것
- **Service requester / consumer (서비스 요청자/소비자)**
    - 다양한 **검색(find) 작업**을 통해 **브로커 레지스트리**에서 항목을 찾고, 서비스 제공자와 **바인딩하여 해당 웹 서비스를 호출**합니다.

## 개요

- SOA는 비즈니스 프로세스에서 시작된다.
→ 여러 다른 서비스들을 붙여가면서 쓰는거임(여러 업무들을 웹서비스와 하는거라서 마치 아마존!)
    - 서비스는 **명확하게 정의된(well-defined)**, **자체적으로 완결된(self-contained)**, **다른 서비스와 독립적인(independent)** 비즈니스 기능입니다.
    - 표준 프로그래밍 인터페이스를 통해 사용됩니다.
- 서비스 지향의 느슨한 결합(loose coupling)은 기업에 유연성(flexibility)을 제공합니다.
    - 이를 통해 **플랫폼 및 기술 제약에 상관없이** 모든 서비스 리소스를 활용할 수 있습니다.

- client는 **service directory**를 통해 service를 찾을 수 있으며 서비스에 **요청-응답(request-response) 방식**으로 접근할 수 있다.    
    사용자가 필요한 서비스를 가져다가 쓰는 것임
    service directory는 클라이언트가 필요한 서비스의 정보를 검색할 수 있도록 도와줌
    

## 구조

- **일반적인 SOA 애플리케이션**은 여러 **가용 서비스**를 활용하며, 이를 흐름 제어 언어(flow control language)를 통해 관리합니다.
    - BPEL for web services → <태그>를 사용
- **Orchestration languages**는 **비즈니스 로직에 따라 실행의 순서와 논리적 순서를 지정**할 수 있도록 합니다.
→ **Orchestration**가 나오면 SOA구나!!!!!!!!!!!

## SOA의 접근방식 구현

- 서비스 지향 아키텍처는 **웹서비스** 또는 **마이크로서비스**(작은 버전)로 구현될 수 있다.
    - 이는 표준 인터넷 프로토콜을 통해 접근할 수 있는 **기능적 빌딩 블록**으로, **플랫폼과 프로그래밍 언어에 독립적**입니다.
- **SOA 구현자**들은 일반적으로 **웹 서비스 표준**을 사용하여 SOA를 구축합니다.
- **SOAP**(Simple Object Access Protocol)이 그 예입니다.
    - 2003년에 W3C (World Wide Web Consortium)에 의해 정의되었습니다.
- SOA는 다른 **서비스 기반 기술**을 사용하여도 구현할 수 있습니다.
    - **Jini, CORBA, Internet Communications Engine, REST, 또는 gRPC** 등이 이에 해당합니다.

- **Service Composition (서비스 구성)**
여러 개의 독립적인 서비스들을 조합해서 더 복잡한 비즈니스 프로세스를 진행한다.

- Service reuse (서비스 재사용)
기존에 구축된 서비스를 다른 서비스나 애플리케이션에서 재사용이 가능하다

- Service composition model (서비스 구성 모델)
Aggregation(집합) 모델: 여러 개의 독립된 서비스를 하나의 새로운 서비스로 조합
Containment(포함) 모델: 외부 서비스가 내부 서비스를 포함하여 사용
→ 내부 서비스는 외부의 제어하에 동작함

## SOA: Advantages

- **Loosely-coupled connections → 여러 서비스가 독립적으로 운영되는 것**
    - **Loose-coupling** 은 SOA에서 중요한 특성이다.
    - 각 서비스 컴포넌트는 상태가 없는(**stateless)** 특성때문에 **독립적**이다.
    - **공개된 인터페이스**가 변경되지 않는 한 서비스의 구현은 애플리케이션에 영향을 주지 않습니다. → 이를 통해 SOA 소프트웨어는 **더 쉽게 진화하고 업데이트** 할 수 있습니다.
- **Interoperability(상호운용성) → 7단계에서 상호호환? 하고 있음**
    - **기술적으로** 모든 클라이언트나 서비스는 **플랫폼, 기술, 공급업체 또는 언어 구현에 상관없이** 다른 서비스에 접근할 수 있습니다.
- **Reusability(재사용성)**
    - **모든 서비스**는 다른 서비스에 의해 **재사용**될 수 있습니다.
    - 클라이언트는 서비스의 **공개 인터페이스**만 알면 되므로, **서비스 구성과 통합**이 훨씬 용이합니다.→ 이를 통해 SOA 기반 비즈니스 애플리케이션 개발은 **시간과 비용 면에서 더 효율적**이 됩니다.
- **Scalability(확장성)**
    - **느슨하게 결합된 서비스**는 **확장하기 쉽습니다**.
    - **문서 지향적**이며 **비동기적인 서비스 기능** 덕분에 확장성을 갖추는 것이 용이합니다.
